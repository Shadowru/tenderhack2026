# Путь создания системы умного поиска

## Хронология разработки

### Этап 0. Планирование архитектуры

**Задача**: спроектировать систему персонализированного поиска для Портала поставщиков zakupki.mos.ru.

**Решения**:
- Docker Compose: 6 сервисов (frontend, backend, Elasticsearch, PostgreSQL, Redis, Ollama)
- FastAPI + React + Tailwind
- Elasticsearch — морфология, fuzzy, синонимы, BM25 + function_score
- Redis — профили пользователей, real-time трекинг
- PostgreSQL — события, контракты, метрики
- Caddy — reverse proxy с автоматическим HTTPS на tenderhack.extra.moscow

**Почему эти технологии**:
- ES выбран за нативную поддержку русской морфологии (stemmer, stop words), fuzzy-запросов и dense_vector
- Redis — для sub-millisecond доступа к профилям при каждом поисковом запросе
- Ollama — для будущих эмбеддингов на CPU без GPU-зависимости

**Ключевые решения**:
- ES маппинг с двумя анализаторами: `russian_custom` (индексация) и `russian_with_synonyms` (поиск) — синонимы раскрываются query-time
- function_score: BM25 × (popularity + product_boosts + category_boosts)
- Трекинг событий с весами: search=1, click=3, cart=5, purchase=10
- Temporal decay: DECAY_FACTOR=0.95 на каждый день давности

### Этап 5. Subject extraction — решение проблемы «тачка с ручкой»

**Проблема**: поиск «ручка» находил «тачка с ручкой», «дверь с ручкой» и другие нерелевантные товары, где «ручка» — атрибут, а не предмет.

**Решение**: морфологический анализ через pymorphy3:
1. Для каждого названия товара определяем часть речи и падеж каждого слова
2. Существительные в именительном падеже → subject (главный предмет)
3. После предлогов (с, для, в, на...) → зависимые, игнорируем
4. Сокращения (руч., табл., бл.) → фильтруем по точке в оригинале
5. Subject сохраняется в отдельное поле ES с максимальным бустом (^5)

**Примеры**:
| Название | Subject |
|----------|---------|
| Ручка шариковая синяя | ручка |
| Тачка строительная с ручкой | тачка |
| Кран шаровый STI руч. в/в 32 | кран |
| Парацетамол табл. 500 мг | парацетамол |
| Набор реагентов для определения | набор реагент |

**Итерации**:
- v1: базовый парсинг → проблема: «руч.» (сокращение) попадало в subject
- v2: добавлена проверка точки после слова → «кран» вместо «руч»
- v3: стоп после бренда (латиницы) → чистые subjects без мусора из артикулов

### Этап 6. Оптимизация поиска: prefix + fuzzy стратегия

**Проблема**: короткие запросы (< 5 символов) с fuzzy давали мусор. «Руч» с fuzziness=1 матчил «руб», «рук», «рус».

**Решение**: двухрежимный поиск:

| Режим | Условие | Стратегия |
|-------|---------|-----------|
| Short (≤ 6 символов, 1 слово) | `match_phrase_prefix` на subject.raw + fuzzy на subject | Prefix для автокомплита + fuzzy для опечаток, только по subject |
| Full (длинный или многословный) | `multi_match` fuzzy по всем полям + should бусты subject.exact | Полнотекстовый поиск с fuzzy + точное совпадение subject |

**subject.exact boost** — keyword subfield, позволяет term/fuzzy query по точному значению subject. При запросе «карандаш» товар с subject=`карандаш` получает буст +15 по сравнению с subject=`клей карандаш`.

**Итерации**:
- v1: prefix на name → проблема: «Кран шаровый руч.» на первом месте
- v2: prefix только на subject → проблема: стемминг «руч» матчил «ручной»
- v3: prefix на subject.raw (без стемминга) → чистые результаты
- v4: добавлен fuzzy на subject.exact для опечаток → «Карндаш» → «Карандаш»

### Этап 7. Убрали explain: true — 10x ускорение

**Проблема**: каждый запрос занимал 1-2 секунды из-за `explain: true` в ES.

**Решение**: убрали explain, reasons строятся из данных документа (текстовое совпадение проверяется in-Python за O(1)).

**Результат**: 30-250 мс вместо 1-2 с.

### Этап 8. Персонализация через историю закупок (cold start)

**Проблема**: новый пользователь приходит с пустым профилем в Redis — персонализации нет.

**Решение**:
1. При загрузке данных: `buyer_ste_counts` (buyer_inn → {ste_id: count}) из контрактов → join с ste_id→category из ES индекса → `buyer_category_weights` в PG
2. При первом поиске: `load_buyer_history(user_id)` — достаём профиль из PG → пушим в Redis sorted set
3. Профиль влияет на ранжирование через category boosts (топ-5 категорий, вес 1.0–1.05)

**Калибровка**: первоначально category_boosts были [1.0, 1.5] → школа видела шлифовальную бумагу на первом месте. Уменьшили до [1.0, 1.05] + ограничили до топ-5 категорий → персонализация стала мягким nudge, а не доминированием.

Реализованы и вычисляются из search_events:

## Архитектурные решения и trade-offs

| Решение | Альтернатива | Почему так |
|---------|-------------|------------|
| ES для полнотекста + Redis для профилей | Всё в ES | Redis даёт sub-ms доступ к профилям; ES не подходит для sorted sets |
| Subject extraction в Python при индексации | NLP-пайплайн в ES | pymorphy3 даёт лемматизацию + падежи, ES stemmer только отсекает окончания |
| Category boosts [1.0, 1.05] | [1.0, 2.0] | Сильные бусты нарушали текстовую релевантность — «шлифовальная бумага» выше «офисной» |
| Top-5 категорий в бустах | Все категории | 420 категорий у школы давали мусор; 5 достаточно для мягкого nudge |
| Spell checker через rapidfuzz (490k словарь) | ES did-you-mean | Работает до ES запроса, ловит больше ошибок (WRatio vs edit distance) |
| Два режима поиска (short/full) | Единый multi_match | Короткие запросы + fuzzy = мусор; prefix решает проблему автокомплита |
| Reasons из данных, не из explain | ES explain: true | explain добавлял 1-2 секунды; reasons из source достаточно информативны |
| Cold start из PG (buyer_category_weights) | Запрос контрактов на лету | Преагрегация при загрузке — O(1) при поиске вместо O(n) join |

## Что не реализовано и почему

| Фича | Причина | Сложность |
|------|---------|-----------|
| Семантические эмбеддинги (Ollama) | CPU-генерация 537k эмбеддингов ~ 24ч; приоритет на текстовом поиске | P2 |
| A/B тестирование персонализации | Нужна инфраструктура для splitting трафика | P2 |
| Авто-пополнение синонимов из кликов | Нужен стат. анализ co-click паттернов | P2 |
| Session-based re-ranking | Нужен session_id + event aggregation в реальном времени | P2 |

---

## Уроки

1. **Персонализация должна быть мягкой**: агрессивные бусты ломают текстовую релевантность. Лучше 5% nudge, чем 50% override.

2. **Subject extraction — ключевая инновация**: разделение «что это» от «что рядом» через морфологический анализ радикально улучшает поиск по коротким запросам.

3. **explain: true убивает производительность**: на 500k индексе добавляет 10x latency. Reasons можно строить из данных.

4. **Spell checker словарь нужно фильтровать**: 490k терминов включают мусор (единицы измерения, артикулы). Фильтр len ≥ 4 решает проблему.

5. **Cold start через преагрегацию**: join контрактов с СТЕ при загрузке → O(1) при первом поиске пользователя. Без этого первый поиск = 2-5 секунд ожидания.

## Этап 11. RAG-powered AI expansion

**Проб��ема**: нечёткие запросы типа «медицинские перчатки» находили «Медицинские услуги» вместо перчаток.

**Решение (RAG-подход)**:
1. ES aggregation без fuzzy → реальные категории из индекса
2. LLM (Qwen 2.5 7B через Ollama) выбирает лучшие категории + генерирует поисковые термины
3. Filtered ES search по каждой выбранной категории
4. Fallback без LLM: топ-3 категории + исходный запрос

**Результат**: «медицинские перчатки» → перчатки нитриловые, хирургические, латексные из категории «Перчатки смотровые/процедурные».

## Этап 12. Phrase match и массовое тестирование

134 запроса (канцелярия, медицина, хозтовары, стройка, услуги, опечатки, prefix, раскладка).

**Проблема**: «туалетная бумага» → SvetoCopy (best_fields брал лучший скор по одному полю «бумага»).

**Решение**: добавлены два should-clause для многословных запросов:
- `match_phrase` с `slop: 3, boost: 12` — ловит перестановки «туалетная бумага» ↔ «бумага туалетная»
- `match` с `operator: "and", boost: 6` — все слова должны быть в name

**Результат**: 89% → **93%** (125/134 запросо�� — релевантный результат в топ-5).

## Уроки (дополнение)

6. **Phrase match критичен для многословных запросов**: best_fields хорош для одиночных слов, но теряет контекст фразы. match_phrase с slop компенсирует.

7. **RAG > слепой LLM**: LLM без контекста галлюцинирует «Канцлерская лампа» и «Клинтоны для врачей». С реальными категориями из индекса — точные результаты.

8. **Тестирование 100+ запросов обязательно**: без массового тестирования «туалетная бумага» → SvetoCopy не была ��ы найдена. Каждая категория запросов (опечатки, prefix, phrase, услуги) требует отдельных тестов.
