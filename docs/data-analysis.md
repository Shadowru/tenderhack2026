### 1.3. Обнаруженные проблемы

#### ВЫСОКИЙ приоритет

**Дубликаты id** — 5 679 пар (11 358 строк), где один и тот же id встречается дважды.
- Решение: при индексации берём первое вхождение, дубль отбрасываем.
- Влияние: 2.1% строк.

**Точные дубликаты по name** — 29 126 групп (89 875 строк, 16.5% от всех).
- Примеры: «Техническая инвентаризация» — 2 198 раз, «Охранные услуги» — 652, «Производственный контроль» — 536.
- Причина: один и тот же товар/услуга с разными id (вероятно, от разных поставщиков или из разных справочников).
- Решение: оставляем все — у каждого свой id и разные спецификации. При поиске ES сам ранжирует.

#### СРЕДНИЙ приоритет

**Нечёткие дубликаты по name** — ещё 2 826 групп, отличающихся регистром/пунктуацией.
- Пример: «Бумага офисная А4» vs «БУМАГА ОФИСНАЯ А4».
- Решение: нормализация при поиске (lowercase + stemming) уже решает эту проблему.

#### НИЗКИЙ приоритет

| Проблема | Кол-во | Решение |
|----------|--------|---------|
| NUL-байты и символ \x02 | 4 строки | Удаление ctrl-символов при загрузке |
| Табуляции в конце name | 2 строки | strip() при загрузке |
| Name длиннее 500 символов | 92 строки | Оставляем как есть, ES индексирует |
| Specs длиной 65 535 | Несколько строк | Вероятно обрезка на стороне источника |

## 2. Контракты

### 2.1. Структура

Формат: CSV, разделитель `;`, кавычки `"`, кодировка UTF-8 с BOM, окончания строк CRLF.

| Поле | Тип | Описание |
|------|-----|----------|
| [0] name | text | Наименование закупки |
| [1] contract_id | integer | ID контракта |
| [2] ste_id | integer | ID СТЕ (внешний ключ) |
| [3] date | timestamp | Дата контракта (YYYY-MM-DD HH:MM:SS.mmm) |
| [4] price | decimal | Цена (руб.) |
| [5] buyer_inn | varchar(10/12) | ИНН заказчика |
| [6] buyer_name | text | Название заказчика |
| [7] buyer_region | text | Регион заказчика |
| [8] supplier_inn | varchar(10/12) | ИНН поставщика |
| [9] supplier_name | text | Название поставщика |
| [10] supplier_region | text | Регион поставщика |

### 2.2. Общая статистика

| Показатель | Значение |
|------------|----------|
| Строк | 2 010 224 |
| Уникальных contract_id | 653 615 |
| Уникальных ste_id | 498 370 |
| Уникальных заказчиков (buyer_inn) | 4 192 |
| Уникальных поставщиков (supplier_inn) | 881 |
| Уникальных регионов заказчиков | 30 |
| Уникальных регионов поставщиков | 42 |

**Примечание**: один contract_id может содержать несколько СТЕ (строк). Поэтому 2 млн строк — это позиции контрактов, а не уникальные контракты.

### 2.3. Обнаруженные проблемы

#### ВЫСОКИЙ приоритет

**Полные дубликаты строк** — 767 строк, где все 11 полей идентичны.
- Решение: дедупликация по ключу (contract_id, ste_id) при загрузке.
- После дедупликации загружено: **2 009 457** уникальных строк.

#### СРЕДНИЙ приоритет

**Цены < 1 руб.** — 246 позиций.
- Примеры: 0.01, 0.10, 0.50 руб.
- Вероятно: символические контракты, тестовые данные или цена за единицу при крупной поставке.
- Решение: оставляем, но не используем для расчёта средних цен.

**Цены > 1 млрд руб.** — 2 позиции (1.576 и 1.005 млрд).
- Вероятно: реальные крупные контракты (строительство, оборудование).
- Решение: оставляем.

**Непечатаемые символы в name** — 550 строк.
- Решение: очистка regex `[\x00-\x08\x0b\x0c\x0e-\x1f]` при загрузке.

#### НИЗКИЙ приоритет

| Проблема | Кол-во | Решение |
|----------|--------|---------|
| Name < 5 символов | 185 | Оставляем ('Соль', 'АМР', '1 шт' — валидные) |
| BOM в начале файла | 1 | Кодировка utf-8-sig при чтении |
| Формат дат с миллисекундами | Все | Парсинг через `%Y-%m-%d %H:%M:%S.%f` |

## 4. Действия при загрузке

При индексации данных применены следующие меры очистки:

1. **Дедупликация СТЕ по id** → 537 314 уникальных (из 542 993)
2. **Дедупликация контрактов по (contract_id, ste_id)** → 2 009 457 (из 2 010 224)
3. **Очистка непечатаемых символов** в name (NUL, STX, табуляции)
4. **Парсинг спецификаций**: извлечение пар key:value, удаление нулевых значений (0, 0.00000)
5. **Расчёт popularity**: на основе количества контрактов, нормализация в [0, 100]
6. **Парсинг дат**: формат timestamp с миллисекундами

## 5. Рекомендации по дальнейшему улучшению

1. **Синонимы из данных**: проанализировать запросы с кликами на одни и те же СТЕ для автоматического пополнения словаря синонимов
2. **Кластеризация дубликатов name**: объединить 29 126 групп дублей в канонические записи для снижения шума
3. **Нормализация категорий**: некоторые категории отличаются пунктуацией/регистром — унифицировать
4. **Валидация ste_id в контрактах**: проверить что все ste_id контрактов существуют в справочнике СТЕ
5. **Временной анализ**: использовать даты контрактов для time-decay в персонализации (свежие закупки весят больше)
