# Подходы к семантическому и морфологическому поиску

## 1. Морфологический поиск

### 1.1. Проблема

Русский язык — флективный: одно слово может иметь десятки словоформ. Запрос «карандаши чернографитные» не найдёт документ с текстом «карандаш чернографитный», если искать точное совпадение строк.

### 1.2. Решение: цепочка анализаторов Elasticsearch

Пайплайн обработки текста приводит и документы, и запросы к единому нормализованному виду:

```
Входной текст
  → Standard Tokenizer (разбиение на слова)
    → Lowercase Filter (приведение к нижнему регистру)
      → Russian Stop Words Filter (удаление предлогов, союзов, частиц)
        → Russian Stemmer (стемминг — отсечение окончаний)
```

### 1.3. Два анализатора

| Назначение | Анализатор | Особенность |
|------------|-----------|-------------|
| Индексация документов | `russian_custom` | Стемминг без синонимов |
| Обработка поискового запроса | `russian_with_synonyms` | Стемминг + раскрытие синонимов |

Синонимы только на этапе поиска — обновлённый файл вступает в силу без переиндексации.

### 1.4. Морфология через pymorphy3

На стороне бэкенда работает **pymorphy3** для:
- **Лемматизации** запросов (начальная форма, не просто отсечение окончания)
- **Определения части речи и падежа** — критично для subject extraction
- **Наполнения словаря** терминов для проверки опечаток (490 204 термина)

## 2. Subject Extraction — ключевая инновация

### 2.1. Проблема

Поиск «ручка» находил «тачка с ручкой», «дверь с ручкой» — товары, где «ручка» является атрибутом, а не предметом.

### 2.2. Решение

Для каждого товара при индексации извлекается **главное существительное (subject)** через морфологический анализ:

1. Проход по словам названия, определение части речи и падежа (pymorphy3)
2. Существительные в **именительном падеже** → subject
3. После предлогов (с, для, в, на...) → зависимые, игнорируются
4. **Сокращения** (руч., табл., бл.) → фильтруются по точке в оригинале
5. После бренда (латиница) → стоп

| Название | Subject |
|----------|---------|
| Ручка шариковая синяя | ручка |
| Тачка строительная с ручкой | тачка |
| Кран шаровый STI руч. в/в 32 | кран |
| Парацетамол табл. 500 мг | парацетамол |
| Набор реагентов для определения | набор реагент |

### 2.3. Использование в поиске

Subject индексируется в отдельное поле ES с тремя sub-fields:

| Поле | Тип | Назначение |
|------|-----|-----------|
| `subject` | text (russian_custom) | Стеммированный поиск |
| `subject.raw` | text (standard) | Prefix-поиск без стемминга |
| `subject.exact` | keyword | Точное совпадение для бустинга |

Subject получает максимальный буст (×5) при ранжировании — товар, где запрос является предметом, всегда выше товара, где он лишь атрибут.

## 3. Двухрежимный поиск

### 3.1. Проблема

Короткие запросы (< 6 символов) с fuzzy давали мусор: «Руч» с fuzziness=1 матчил «руб», «рук», «рус».

### 3.2. Решение: два режима

| Режим | Условие | Стратегии |
|-------|---------|-----------|
| **Short** | 1 слово, ≤ 6 символов | `match_phrase_prefix` на subject.raw + fuzzy на subject |
| **Full** | длинный или многословный | 4 стратегии в `bool.should` (см. ниже) |

### 3.3. Стратегии полного поиска (should)

Для многословных запросов используются 4 параллельных стратегии, ES берёт лучший результат:

1. **best_fields fuzzy** — классический multi_match по 7 полям с fuzziness:AUTO
2. **match_phrase slop:3** — бустит фразы с перестановками («туалетная бумага» ↔ «бумага туалетная»)
3. **match_phrase slop:1** — бустит точные числовые последовательности («100 листов» > «500 листов»)
4. **match AND** — все слова запроса должны присутствовать в name (boost 15)

### 3.4. Subject exact boost

Дополнительный should-бустинг по `subject.exact` (keyword):
- Точное совпадение леммы запроса с subject → +15 boost
- Fuzzy-совпадение для опечаток → +12 boost

Результат: «карандаш» → товар с subject=`карандаш` выше чем subject=`клей карандаш`.

## 4. Обработка опечаток и вариативности ввода

### 4.1. Fuzzy-поиск (расстояние Левенштейна)

`fuzziness: "AUTO"` с `prefix_length: 1`:
- 3–5 символов → 1 ошибка
- 6+ символов → 2 ошибки

### 4.2. Spell Checker (rapidfuzz)

Словарь 490 204 термина (названия + категории СТЕ, len ≥ 4). Алгоритм WRatio, threshold 65%.

Примеры: «карндаш» → Карандаш, «бумга» → Бумага, «прнтер» → Принтер.

### 4.3. Коррекция раскладки EN→RU

Детекция строк из латинских символов → конвертация по таблице QWERTY → ЙЦУКЕН.

| Домен | Примеры | Кол-во правил |

| Офис | файл = файл-вкладыш = папка-файл | 12 |

## 6. Персонализация

### 6.1. Принцип

Два заказчика, вводя одинаковый запрос «бумага», получают разные результаты:
- **Школа** → сертификат-бумага, дизайн-бумага (бланки для грамот)
- **Колледж** → тарелка из ламинированной бумаги (столовая), ролики для принтеров
- **Больница** → SvetoCopy Classic A4 (офисная бумага)

### 6.2. Cold Start из контрактов

При первом визите пользователя (по ИНН организации):

1. Таблица `buyer_category_weights` (520 874 записей, 4 190 организаций)
2. Содержит предагрегированные веса категорий из 2 009 457 контрактов
3. При первом поиске → загрузка из PG в Redis sorted set
4. Исторические данные дисконтируются ×0.7 (live-события приоритетнее)

### 6.3. Сбор сигналов в реальном времени

Каждое действие пользователя записывается в Redis с весом:

| Событие | Вес | Обоснование |
|---------|-----|-------------|
| Поиск | 1.0 | Слабый сигнал — пользователь только ищет |
| Просмотр карточки | 2.0 | Открыл детальную карточку товара |
| Клик | 3.0 | Средний — проявил интерес в выдаче |
| Добавление в корзину | 5.0 | Сильный — намерение закупки |
| Покупка | 10.0 | Максимальный — реальная потребность |

**Bounce rate**: если пользователь закрыл карточку товара быстрее 3 секунд → event `quick_return` (негативный сигнал).

### 6.4. Профиль пользователя (Redis)

| Ключ | Тип | Описание |
|------|-----|----------|
| `user:{inn}:categories` | sorted set | Категории с накопленными весами |
| `user:{inn}:products` | sorted set | Товары с накопленными весами |
| `user:{inn}:events` | list (FIFO) | Последние 1000 событий с timestamps |
| `cart:{inn}` | hash | Корзина: product_id → quantity |

TTL: 90 дней. Temporal decay: `DECAY_FACTOR = 0.95` (вес ×0.95 за каждый день давности).

### 6.5. Двухуровневый бустинг

При каждом поисковом запросе:

**Product-level boosts** (топ-50 товаров пользователя):
1. Из Redis → sorted set `products` → топ-50 по весу
2. Decay из event log: `weight × 0.95^(days_ago)`
3. Нормализация в [1.0, 2.0]
4. Каждый товар → `function_score` filter с weight

**Category-level boosts** (топ-5 категорий пользователя):
1. Из Redis → sorted set `categories` → топ-5
2. Нормализация в [1.0, 1.05]
3. Fuzzy match по тексту категории (`minimum_should_match: 75%`) — ловит смежные категории
4. Применяется как `function_score` с `boost_mode: "multiply"`

### 6.6. Персонализация подсказок (suggest)

Автокомплит тоже персонализирован:
- Category boosts амплифицируются ×30 для suggest (иначе слишком слабые)
- `collapse` по `category.keyword` — один товар из каждой категории
- Результат: школа видит учебники, колледж — ролики для принтеров

## 7. AI-расширение поиска (RAG)

### 7.1. Проблема

Нечёткие запросы типа «медицинские перчатки» находили «Медицинские услуги» вместо перчаток.

### 7.2. RAG-подход

Трёхшаговый процесс с использованием LLM (Qwen 2.5 7B через Ollama):

**Шаг 1**: ES aggregation без fuzzy → реальные категории из индекса:
```
"медицинские перчатки" →
  - Перчатки смотровые/процедурные (1086 товаров)
  - Перчатки резиновые общего назначения (897)
  - Медицинские перчатки (50)
```

**Шаг 2**: LLM выбирает 3 лучших категории + генерирует поисковые термины:
```
Перчатки смотровые | перчатки медицинские
Медицинские перчатки | перчатки одноразовые
```

**Шаг 3**: Поиск ES с category filter для каждой выбранной категории → точные результаты.

**Fallback** (если LLM не отвечает): топ-3 категории + исходный запрос.

### 7.3. UI

AI-результаты отображаются в отдельном правом сайдбаре (sticky, collapsible). Запускаются параллельно основному поиску — пользователь видит обычные результаты мгновенно, AI-расширение подгружается через 3-8 секунд.

## 8. Карточка товара и корзина

### 8.1. Карточка товара

При клике на результат поиска открывается модальное окно с:
- Полным наименованием, категорией, subject
- Бейджами: количество закупок, популярность
- Таблицей характеристик (парсятся из specifications)
- Кнопкой «Добавить в корзину»

### 8.2. Bounce rate tracking

При открытии карточки фиксируется timestamp. При закрытии:
- Если < 3 секунд → event `quick_return` (негативный сигнал)
- Если ≥ 3 секунд → event `view` (позитивный сигнал, вес 2.0)

### 8.3. Корзина

Redis-backed корзина (`cart:{inn}` → hash product_id → quantity):
- Добавление в корзину → event `cart` (вес 5.0) → влияет на персонализацию
- Slide-out панель в UI с количеством, удалением, очисткой

## 9. Итоговая формула ранжирования

```
Score = BM25(query, strategies) × Popularity(log2p) × ProductBoost × CategoryBoost
```

Где:

| Компонент | Диапазон | Описание |
|-----------|----------|----------|
| BM25 strategies | 0–∞ | max(fuzzy_multimatch, phrase_slop3, phrase_slop1, AND_match) |
| Popularity | 1.0–3.0 | `log2p(popularity * 0.5)` — мягкий, не доминирует |
| ProductBoost | 1.0–2.0 | Персональный буст по товарам с temporal decay |
| CategoryBoost | 1.0–1.05 | Персональный буст по категориям (top-5) |
| SubjectExact | +15 | Бонус за точное совпадение subject с леммой запроса |
| SubjectFuzzy | +12 | Бонус за fuzzy-совпадение subject (для опечаток) |

## 10. Полный пайплайн обработки запроса

```
Пользователь вводит запрос
  │
  ├─ 1. Нормализация (пробелы, регистр)
  ├─ 2. Детекция и исправление раскладки EN→RU
  ├─ 3. Проверка опечаток (rapidfuzz, 490K словарь → «вы имели в виду?»)
  │
  ├─ 4. Cold start: загрузка профиля из PG → Redis (если первый визит)
  ├─ 5. Загрузка бустов: product_boosts + category_boosts из Redis
  │
  ├─ 6. Определение режима: short (≤6 символов) или full (многословный)
  │
  ├─ 7. Отправка в Elasticsearch:
  │     ├─ Short: prefix на subject.raw + fuzzy на subject
  │     ├─ Full: 4 стратегии (fuzzy + phrase×2 + AND)
  │     ├─ Subject exact/fuzzy boost в should
  │     ├─ Function score: popularity × product_boosts × category_boosts
  │     └─ Агрегация по категориям
  │
  ├─ 8. Параллельно: AI-расширение (RAG через Ollama)
  │     ├─ ES aggregation → категории
  │     ├─ LLM → выбор категорий + поисковые термины
  │     └─ Filtered ES search по каждой категории
  │
  ├─ 9. Формирование ответа: items + reasons + categories + AI expansion
  │
  ├─10. Кэширование: Redis (60с auth / 300с anon / 600с suggest)
  │
  └─11. Трекинг: событие в Redis (sorted sets) + PostgreSQL (search_events)
        ├─ При открытии карточки → event "view" (2.0)
        ├─ При bounce < 3с → event "quick_return"
        └─ При добавлении в корзину → event "cart" (5.0)
```

## 11. Метрики качества

| Метрика | Формула | Цель |
|---------|---------|------|
| MRR | mean(1/позиция первого клика) | > 0.5 |
| Precision@5/10/20 | релевантных в топ-K / K | > 40% |
| CTR | клики / поиски | > 30% |
| Session Success Rate | сессий с кликом / всего | > 50% |
| Средняя позиция клика | avg(position при клике) | < 3.0 |
| Personalization Lift | CTR_персонализ / CTR_базовый - 1 | > 10% |

**Результат тестирования**: 93% (125/134 запросов) — релевантный результат в топ-5.
