# Подходы к семантическому и морфологическому поиску

## 1. Морфологический поиск

### 1.1. Проблема

Русский язык — флективный: одно слово может иметь десятки словоформ. Запрос «карандаши чернографитные» не найдёт документ с текстом «карандаш чернографитный», если искать точное совпадение строк.

### 1.2. Решение: цепочка анализаторов Elasticsearch

Мы строим пайплайн обработки текста, который приводит и документы, и запросы к единому нормализованному виду:

```
Входной текст
  → Standard Tokenizer (разбиение на слова)
    → Lowercase Filter (приведение к нижнему регистру)
      → Russian Stop Words Filter (удаление предлогов, союзов, частиц)
        → Russian Stemmer (стемминг — отсечение окончаний)
```

**Пример работы цепочки:**

| Этап | Результат |
|------|-----------|
| Исходный текст | «Карандаши чернографитные простые» |
| Tokenizer | `Карандаши`, `чернографитные`, `простые` |
| Lowercase | `карандаши`, `чернографитные`, `простые` |
| Stop words | `карандаши`, `чернографитные`, `простые` |
| Stemmer | `каранд`, `чернографитн`, `прост` |

Теперь запрос «карандаш простой» пройдёт ту же цепочку и даст стемы `каранд`, `прост` — которые совпадут с индексированными.

### 1.3. Два анализатора — зачем

Мы используем **разные** анализаторы для индексации и поиска:

| Назначение | Анализатор | Особенность |
|------------|-----------|-------------|
| Индексация документов | `russian_custom` | Стемминг без синонимов |
| Обработка поискового запроса | `russian_with_synonyms` | Стемминг + раскрытие синонимов |

Почему синонимы только на этапе поиска? Если раскрывать синонимы при индексации, индекс разбухает и обновление синонимов требует полной переиндексации. При подходе query-time синонимы раскрываются на лету, и обновлённый файл синонимов вступает в силу сразу.

### 1.4. Дополнительная морфология через pymorphy3

Помимо ES-стеммера, на стороне бэкенда работает **pymorphy3** — полноценный морфологический анализатор русского языка. Он используется для:

- **Лемматизации** запросов (приведение к начальной форме словаря, а не просто отсечение окончания)
- **Наполнения словаря** терминов для проверки опечаток

Разница между стеммингом и лемматизацией:

| Слово | Стемминг (ES) | Лемматизация (pymorphy3) |
|-------|--------------|--------------------------|
| бегущий | `бег` | `бежать` |
| карандашей | `каранд` | `карандаш` |
| лучше | `лучш` | `хороший` |

Стемминг быстрее и достаточен для поиска (совпадение стемов). Лемматизация точнее и используется для проверки опечаток и подсказок.

## 2. Обработка опечаток и вариативности ввода

### 2.1. Fuzzy-поиск (расстояние Левенштейна)

Elasticsearch `multi_match` с параметром `fuzziness: "AUTO"`:

- Слова до 2 символов — точное совпадение
- 3–5 символов — допускается 1 ошибка
- 6+ символов — допускается 2 ошибки

Параметр `prefix_length: 2` требует, чтобы первые 2 символа совпадали точно — это предотвращает слишком широкий поиск и повышает производительность.

**Пример:** запрос `каранадш` (опечатка) найдёт `карандаш` (расстояние = 1 транспозиция).

### 2.2. Коррекция раскладки клавиатуры

Частая ошибка: пользователь начинает набирать текст, не переключив раскладку с английской. Запрос `,evfuf` — это «бумага» на QWERTY-клавиатуре.

Система детектирует, что строка состоит только из латинских символов, и конвертирует её по таблице соответствия клавиш EN→RU. Конвертация происходит **до** отправки в Elasticsearch.

### 2.3. Fuzzy-matching по словарю (rapidfuzz)

Помимо встроенного fuzzy в ES, на бэкенде работает отдельная проверка через `rapidfuzz` (алгоритм WRatio — взвешенное соотношение):

1. Берётся словарь известных наименований (из каталога СТЕ)
2. Запрос пользователя сравнивается со словарём
3. Если найдено совпадение ≥ 75% (и оно отличается от исходного запроса) — пользователю предлагается «Возможно, вы имели в виду: ...»

Это **не автозамена**, а подсказка — пользователь сам решает, принять ли исправление.

## 3. Система синонимов

### 3.1. Файл синонимов

Elasticsearch загружает файл `synonyms.txt` с правилами эквивалентности:

```
бумага а4, бумага формата а4, бумага офисная а4
принтер, печатающее устройство, мфу
компьютер, пк, персональный компьютер, эвм
```

Формат: все термины через запятую считаются эквивалентными. Запрос по любому из них найдёт документы с любым другим.

### 3.2. Динамическое обновление

Фильтр синонимов настроен с флагом `updateable: true`. Это позволяет обновлять файл и перезагружать анализатор без полной переиндексации — через API `_reload_search_analyzers`.

### 3.3. Стратегия наполнения

Синонимы собираются из нескольких источников:

1. **Экспертные** — ручное добавление стандартных сокращений и эквивалентов (ПК = персональный компьютер)
2. **Из данных** — анализ запросов, которые приводят к кликам на одни и те же товары (если «мфу» и «принтер» ведут к одним карточкам — это синонимы)
3. **Из пустых выдач** — запросы с нулём результатов проверяются на близость к существующим терминам

## 4. Семантический поиск (векторный)

### 4.1. Идея

Морфология и синонимы решают проблему словоформ и известных эквивалентов. Но они не помогут, если пользователь ищет «средство для письма», а в каталоге — «ручка шариковая». Слова разные, стемы разные, в словаре синонимов такой пары нет.

Семантический поиск решает эту задачу: текст превращается в числовой вектор (эмбеддинг), где близкие по смыслу тексты оказываются рядом в пространстве.

### 4.2. Модель эмбеддингов

Используем **Ollama** с моделью `nomic-embed-text` (768 измерений):

- Работает на CPU (не требует GPU)
- Поддерживает русский язык
- Компактная и быстрая для inference

Каждый товар в каталоге индексируется как текст → вектор 768d, который сохраняется в поле `embedding` (тип `dense_vector`) в Elasticsearch.

### 4.3. Cosine similarity

Elasticsearch поддерживает векторный поиск через `knn` или `script_score`. Мы используем **cosine similarity** — мера близости двух векторов от 0 до 1:

- 1.0 — идентичные по смыслу
- 0.0 — никакой связи

При поиске запрос также превращается в вектор и сравнивается со всеми документами. Elasticsearch использует HNSW-индекс для быстрого приближённого поиска (не перебирает все документы).

### 4.4. Ограничения

- **Latency**: генерация эмбеддинга запроса на CPU занимает 50–200 мс (добавляется к времени поиска)
- **Качество**: модели общего назначения не всегда хорошо понимают специфику госзакупок (СТЕ, ОКПД2 и т.д.)
- **Объём**: 768 float на документ ≈ 3 КБ; при 1 млн документов ≈ 3 ГБ только на векторы

## 5. Гибридный поиск: объединение подходов

### 5.1. Архитектура ранжирования

Итоговое ранжирование использует **function_score** в Elasticsearch, объединяя несколько сигналов:

```
Итоговый скор = BM25(запрос) × (популярность + персонализация)
```

Где:

| Компонент | Что делает | Вес |
|-----------|-----------|-----|
| BM25 multi_match | Классический текстовый скор с учётом морфологии, синонимов, fuzzy | Базовый |
| popularity (field_value_factor) | Глобальная популярность товара (log1p, factor 1.2) | Множитель |
| Персональные бусты | Веса конкретных товаров для данного пользователя [1.0–2.0] | Множитель |

### 5.2. Веса полей

Не все поля документа одинаково важны:

| Поле | Буст | Почему |
|------|------|--------|
| `name` | ×3 | Наименование — главный индикатор релевантности |
| `category` | ×2 | Категория сужает область поиска |
| `description` | ×1.5 | Описание содержит дополнительный контекст |
| `specifications` | ×1 | Характеристики — дополнительный сигнал |
| `name.ngram` | ×1 | Частичные совпадения (для коротких запросов) |

### 5.3. Порог совпадения

Параметр `minimum_should_match: "70%"` означает: если в запросе 3 слова, должны совпасть хотя бы 2. Это баланс между полнотой (recall) и точностью (precision):

- 100% — слишком строго, теряем результаты при опечатке в одном слове
- 50% — слишком мягко, много мусора в выдаче
- 70% — оптимум для запросов из 2–5 слов

## 6. Персонализация: динамическая переиндексация выдачи

### 6.1. Принцип

Два заказчика, вводя одинаковый запрос «бумага», ожидают разные результаты:
- Школа → бумага для рисования, цветная бумага
- Офис → бумага А4, бумага для принтера

Система запоминает историю взаимодействия каждого пользователя и **подстраивает ранжирование** под его профиль.

### 6.2. Сбор сигналов

Каждое действие пользователя записывается с весом:

| Событие | Вес | Обоснование |
|---------|-----|-------------|
| Поиск | 1.0 | Слабый сигнал — пользователь только ищет |
| Клик | 3.0 | Средний — проявил интерес |
| Добавление в корзину | 5.0 | Сильный — намерение закупки |
| Покупка | 10.0 | Максимальный — реальная потребность |

### 6.3. Профиль пользователя (Redis)

Для каждого пользователя в Redis хранятся sorted sets:

- `user:{id}:categories` — категории с накопленными весами
- `user:{id}:products` — товары с накопленными весами
- `user:{id}:events` — последние 1000 событий (FIFO)

TTL: 90 дней. Затухание: `DECAY_FACTOR = 0.95` (вес уменьшается на 5% в день).

### 6.4. Бустинг в реальном времени

При каждом поисковом запросе:

1. Из Redis достаются топ-50 товаров пользователя с весами
2. Веса нормализуются в диапазон [1.0, 2.0]
3. Для каждого товара добавляется `function_score` фильтр
4. Elasticsearch умножает BM25-скор на персональный буст

Таким образом, если пользователь раньше покупал определённые товары, они поднимутся в выдаче при похожих запросах — без изменения самого индекса.

## 7. Итоговый пайплайн обработки запроса

```
Пользователь вводит запрос
  │
  ├─ 1. Нормализация (пробелы, регистр)
  ├─ 2. Детекция и исправление раскладки EN→RU
  ├─ 3. Проверка опечаток (rapidfuzz → подсказка «вы имели в виду?»)
  │
  ├─ 4. Загрузка персональных бустов из Redis
  │
  ├─ 5. Отправка в Elasticsearch:
  │     ├─ Токенизация → lowercase → синонимы → стемминг
  │     ├─ Multi-match по полям с весами + fuzzy
  │     ├─ Function score: популярность + персонализация
  │     └─ Агрегация по категориям
  │
  ├─ 6. Формирование ответа с highlight'ами
  │
  └─ 7. Асинхронная запись события в Redis + PostgreSQL
```
